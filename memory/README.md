# Memory — Layer 3 of the Harness

Memory is what Claude remembers about you across sessions.
Mostly you don't configure this — you just need to know how it works
so you can use it deliberately.

---

## Where It Lives

Claude Code's auto-memory lives at:

```
~/.claude/projects/<your-home-path>/memory/
```

This directory contains:

- `MEMORY.md` — an always-loaded index (keep it under ~200 lines).
- Individual topic files — one per area of knowledge.

You do NOT need to create this directory yourself. Claude Code
initializes it when you first ask Claude to remember something.

---

## Four Types of Memory Worth Keeping

| Type          | What                                  | Example                                              |
| ------------- | ------------------------------------- | ---------------------------------------------------- |
| **User**      | Who you are, how you work             | "Senior backend dev, 10 years Python, new to Rust"   |
| **Feedback**  | Corrections and validated approaches  | "Use real DB in integration tests, not mocks"        |
| **Project**   | Current state of what you're building | "Migration launched 2026-04-22, 50k rows backfilled" |
| **Reference** | Where to find external information    | "Bugs tracked in Linear project INGEST"              |

---

## What NOT to Save in Memory

Memory should not duplicate anything that already lives in:

- **Code** (the code is the source of truth, don't re-document it).
- **Git history** (`git log` / `git blame` are authoritative).
- **Your project CLAUDE.md** (Layer 4).
- **Your rules** (Layer 2).

And never save:

- Secrets, credentials, or tokens.
- PII or customer data.
- Ephemeral task state ("working on X right now").

---

## How to Trigger a Save

You don't have to format anything. Just ask:

- "Remember that I prefer X."
- "Save this: we decided to use Postgres, not Mongo, because Y."
- "For future reference, our deploy dashboard is at Z."

Claude writes the memory in the correct type and updates the index.

## How to Trigger a Removal

- "Forget what you know about X."
- "Remove the memory about Y — it's out of date."

---

## Version Control

**Do not commit memory to version control.**

Memory is your local working state. It has personal notes,
half-formed opinions, and context that doesn't belong in a repo.

Add `~/.claude/` to your **global** `.gitignore`:

```bash
git config --global core.excludesfile ~/.gitignore_global
echo ".claude/" >> ~/.gitignore_global
```

---

## What This Directory Does NOT Contain

This `memory/` folder in the starter kit is **just this README**.
There are no memory files to copy. Memory is generated by your own
Claude Code sessions over time — it can't (and shouldn't) be shipped
from one person to another.

The only reason this folder exists in the starter is to make sure
you understand Layer 3 is part of the architecture, even though
there's nothing to install.
